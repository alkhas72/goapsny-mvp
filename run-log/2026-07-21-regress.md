## Сводка: симптом → причина → фикс

### ✅ НАХОДКА 1 (РЕАЛЬНЫЙ БАГ, чиню)

**Симптом:** `https://goapsny-mvp.vercel.app/public.html` отдаёт **HTTP 404 Not Found**, а не SPA-fallback на главный вход. Любая закладка/внешняя ссылка на старый вход сломана.

**Причина:** До П-22 Vercel деплоил `dist/public.html` как реальный статический файл. После П-22 этого файла нет. В репо никогда не было `vercel.json` с rewrites/redirects — Vercel работает с дефолтными настройками (unknown path = 404).

Локально это маскировалось: `vite preview` по умолчанию отдаёт `index.html` на любой путь (SPA mode), поэтому проверка «/public.html отдаёт тот же бандл» в прошлом проходе дала ложный позитив.

**Фикс:** Добавлен `vercel.json` с redirect 301 `/public.html` → `/`:
```json
{
  "$schema": "https://openapi.vercel.sh/vercel.json",
  "redirects": [
    { "source": "/public.html", "destination": "/", "permanent": true }
  ]
}
```

Permanent 301 выбран намеренно:
- Закладки во всех браузерах автоматически обновятся.
- Поисковые индексы переносят ранг на `/`.
- Telegram Mini App menu button / share links, если ссылались на старый URL, откроются корректно.
- Не маскирует будущие реальные 404 (как сделал бы общий SPA rewrite `/* → /index.html`).

Деплойится автоматически при следующем merge в main.

### ⚠️ НАХОДКА 2 (НЕ П-22 REGRESS, фикс вне этой партии)

**Симптом:** `npm test` падает с 1 ошибкой при наличии `.env`:
```
src/services/api.test.ts > api.loginTelegram > throws when Supabase is not configured
AssertionError: expected [Function] to throw error matching /не настроен/i
but got 'Вход через Telegram отклонён: invalid…'
```

**Причина:** Тест ожидает, что `requireConfigured()` бросит `Error('Supabase не настроен...')`. Но при наличии `.env` с реальным production URL, `isSupabaseConfigured()` = true, и код идёт дальше в `fetch()`, который failing with server-side rejection.

**Это test isolation issue, не баг кода.** На CI без `.env` всё зелёно (168/168 PASS). Относится к Telegram-контуру (api.ts), добавлен в PR #10 (Г1).

**Фикс:** вне этой партии. Рекомендация: тест должен явно `vi.stubEnv('VITE_SUPABASE_URL', '')` и `vi.stubEnv('VITE_SUPABASE_ANON_KEY', '')` в начале теста.

### ✅ НЕ П-22 REGRESS (но любопытно)

**Title в `<head>`:** у `/public.html` был `<title>GoApsny — Публичная карта</title>`, у `/index.html` теперь `<title>GoApsny — Карта доступности</title>`. Это **намеренное объединение**, не регрессия. После redirect 301 все посетители увидят унифицированный title.

### ✅ ЛОЖНАЯ ТРЕВОГА (не regs, зафиксировал для честности)

**Симптом:** При одном из прогонов `npm run browser-smoke` локально все viewport'ы показали `error state` с сообщением «Нет конфигурации Supabase».

**Причина:** В процессе поиска я делал `mv .env .env.bak` → `npm test` → `mv .env.bak .env`. Между этими шагами Vite закэшировал состояние без `.env` в `node_modules/.vite/`. Последующий `npm run build` использовал кэш и не подхватил `.env` обратно — бандл оказался без env vars, хотя `.env` физически присутствовал.

**Фикс:** `rm -rf node_modules/.vite dist && npm run build`. Не regs П-22 — production работает (доказано live probe: 1 метка, 0 ошибок).
