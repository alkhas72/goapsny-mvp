## 5. Block B+C — Storage boundary + `submit_public_place` RPC

Migration `20260715120000_submit_public_place_rls_rpc.sql`:

### 5.1 Public-user Storage INSERT (narrow)
Policy `place_photos_storage_insert_public_user_facade` on `storage.objects` `for insert to authenticated`:
```text
bucket_id = 'place-photos'
and owner = (select auth.uid())
and exists (select 1 from public.profiles p where p.id=auth.uid() and p.role='public_user')
and name matches canonical lowercase UUID + '/facade.jpg'
```
- Shape only `{uuid}/facade.jpg` (facade kind), owned by caller, public_user role. The storage layer cannot know the caller's chosen `place_id`, so it enforces the **shape + ownership + role**; the exact place_id↔path binding is RPC-enforced (§5.4). This matches OBJECT-CONTRACT §2 (`id | UUID generated once; first photo-path segment`) — `place_id` is a fresh per-place UUID, **not** the user id.
- No extra kinds, no cross-owner, no update, no list-all. Bucket stays private.

### 5.2 Public-user Storage DELETE (narrow, unreferenced)
Policy `place_photos_storage_delete_public_user_unreferenced` `for delete to authenticated`:
```text
bucket_id = 'place-photos'
and owner = (select auth.uid())
and an explicit matching public_user profile exists
and name matches canonical lowercase UUID + '/facade.jpg'
and goapsny_private.claim_unreferenced_storage_path(name)
```
- Lets a `public_user` remove only their own unreferenced facade orphan after a failed RPC. The non-public SECURITY DEFINER helper takes the same advisory transaction lock as submit, so cleanup cannot race with facade validation/photo indexing and sees hidden references despite caller RLS.

### 5.3 Baseline policy hardening
T2 drops/recreates the baseline Storage UPDATE, authenticated Storage SELECT, place UPDATE and photo UPDATE policies. `public_user` cannot mutate submitted place/photo/Storage rows or list pending/hidden bucket objects. Trusted collectors/admins keep their operational paths; ordinary authenticated users see only facade objects indexed to published places.

### 5.4 RPC `public.submit_public_place`
Signature (only the minimum typed inputs the client may supply; `place_id` is a **fresh client-generated per-place UUID**, per OBJECT-CONTRACT §2):
```sql
create or replace function public.submit_public_place(
  p_place_id uuid,
  p_name text,
  p_category text,
  p_lat double precision,
  p_lng double precision,
  p_storage_path text
) returns public.places
language plpgsql
security definer
set search_path = ''
as $$
declare
  v_uid uuid := auth.uid();
  v_expected_path text;
begin
  -- 1. authenticated
  if v_uid is null then
    raise exception 'submit_public_place: authentication required' using errcode = '42501';
  end if;
  -- 2. active, non-banned public_user profile matching auth.uid()
  if not exists (
    select 1 from public.profiles p
    where p.id = v_uid and p.role = 'public_user'
  ) then
    raise exception 'submit_public_place: caller must be an active public_user' using errcode = '42501';
  end if;
  -- 3. trimmed non-empty name
  if p_name is null or btrim(p_name) = '' then
    raise exception 'submit_public_place: name is required' using errcode = '23502';
  end if;
  -- 4. active category
  if not exists (
    select 1 from public.categories c where c.slug = p_category and c.is_active = true
  ) then
    raise exception 'submit_public_place: inactive or unknown category' using errcode = '23503';
  end if;
  -- 5. finite coordinates within table CHECK bounds
  if p_lat is null or p_lng is null
     or p_lat <> p_lat or p_lng <> p_lng  -- NaN check (NaN <> NaN is true)
     or p_lat < -90 or p_lat > 90 or p_lng < -180 or p_lng > 180 then
    raise exception 'submit_public_place: invalid coordinates' using errcode = '22P02';
  end if;
  -- 6. exact facade path derived from place_id (client cannot forge an arbitrary path)
  v_expected_path := p_place_id::text || '/facade.jpg';
  if coalesce(p_storage_path, '') <> v_expected_path then
    raise exception 'submit_public_place: storage_path must equal {place_id}/facade.jpg' using errcode = '22P02';
  end if;
  -- 7. place_id not already used
  if exists (select 1 from public.places pl where pl.id = p_place_id) then
    raise exception 'submit_public_place: place_id already used' using errcode = '23505';
  end if;
  -- 8. photo path not already indexed
  if exists (select 1 from public.photos ph where ph.storage_path = v_expected_path) then
    raise exception 'submit_public_place: facade already indexed' using errcode = '23505';
  end if;
  -- 9. private Storage object EXISTS and is owned by the caller (server-side revalidation)
  if not exists (
    select 1 from storage.objects so
    where so.bucket_id = 'place-photos'
      and so.name = v_expected_path
      and so.owner = v_uid
  ) then
    raise exception 'submit_public_place: owned facade object not found' using errcode = '23503';
  end if;

  -- 10. atomic insert: place first, then facade photo, single transaction
  insert into public.places (
    id, name, category, lat, lng,
    status, moderation_status, source, created_by, details
  )
  values (
    p_place_id, btrim(p_name), p_category, p_lat, p_lng,
    'gray', 'published', 'public', v_uid, '{"schema_version":1}'::jsonb
  )
  returning 1 into v_inserted_placeholder;   -- (see note; use a boolean flag in real SQL)

  insert into public.photos (place_id, storage_path, kind, uploaded_by)
  values (p_place_id, v_expected_path, 'facade', v_uid);

  select * into v_result from public.places where id = p_place_id;
  return v_result;
end;
$$;
```

All server-hardcoded: `status='gray'`, `moderation_status='published'`, `source='public'`, `created_by=v_uid`, `details='{"schema_version":1}'::jsonb`, photo `kind='facade'`, `uploaded_by=v_uid`. No client-supplied role/creator/status/moderation/source/details/audit/photo-kind. Single function = single transaction → any raised exception rolls back both inserts (no half-object).

> The pseudo-PL above sketches the logic; the migration file uses a clean `declare` block with a `v_done boolean` flag for the `insert ... returning` (no placeholder), a `v_result public.places%rowtype`, and the `NaN <> NaN` idiom is replaced by the equivalent finite check that Postgres accepts. (Postgres `double precision` NaN compares equal to itself under `=`, so the explicit `<` bounds plus `is null` is the reliable guard; I'll use a small `isfinite()`-equivalent via `p_lat::text <> 'NaN'` if needed — finalized in the migration.)

## 7. Decisions to flag (not silently made)

1. **`place_id` is a fresh client-generated per-place UUID** (per OBJECT-CONTRACT §2: `id | UUID generated once; first photo-path segment`). The Storage INSERT policy enforces shape `{uuid}/facade.jpg` + caller ownership + `public_user` role (it cannot know the chosen place_id); the RPC binds the **exact** `{place_id}/facade.jpg` path and revalidates that a private Storage object with that name exists and is owned by `auth.uid()`. This is the only reading consistent with both "exact path `{place_id}/facade.jpg`" (contract §3) and "place id is a fresh per-place UUID" (contract §2). Surfaced in report for transparency; not an open fork.
2. **No backfill of existing rows.** Cannot safely distinguish email-only identities among existing `tester` rows from data alone → omitted per brief A.4, recorded as migration prerequisite.
3. **`current_user_role()` etc. reused, not rewritten.** They are legacy `search_path = public` but read-only helpers; the brief says "do not copy that pattern into new RPCs" and "do not expand T2 into an unrelated rewrite". New RPC uses `search_path = ''`. Reusing the existing helpers as boolean subqueries keeps T2 additive.
