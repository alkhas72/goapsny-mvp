## 1. Правила исполнения

1. **Порядок фаз — по утверждённой партитуре** (`goapsny-epoch3-working-plan-2026-07-17`, approved): объединение оболочек доказывается **отдельно** от замены карты, с проверкой после каждого слоя. Замена карты (MapLibre) — только после контрольной точки G-EQ; удаление Leaflet — только после G1.
2. **Классы агентов (по аудиту AGY):**
   - **Тяжёлые workers:** K3, C1, C2 — сложные продуктовые партии (поведение, auth, state-машины, схема/RLS).
   - **Лёгкие workers:** A1, A2, B1, B2 — механические/локальные партии (переносы, deps, CSS, удаления, test-only).
   - **AGY — аудитор:** независимая приёмка и визуальные гейты; **не автор** ни одной партии.
3. **Дирижёр = Codex.** Лично продукт не исполняет. Единственный commit-proxy интеграционной ветки `feat/epoch3-integrate`: принимает проверенные commits/cherry-picks **строго последовательно**, ведёт очередь влитий, следит за сериализацией горячих файлов (§9). Не автор кода.
4. **Каждая продуктовая партия — worker в отдельном worktree/ветке.** Workers **не коммитят** в интеграционную ветку. Перед сдачей worker делает rebase своей ветки на текущую вершину интеграционной; Дирижёр вливает одну партию за раз.
5. **Партия = 1–3 файла.** Дробление фиксировано ниже и не менялось с прошлой ревизии.
6. **Резервы:** в каждой фазе явно названы один тяжёлый и один лёгкий резерв; резервы ротируются между фазами (§3).
7. **`docs/backend-contract-2026-06-01.md` не меняется без явного согласования с Алхасом** (`AGENTS.md`). П-21 реализует и фиксирует контракт, уже задокументированный в docstring `submit-place.ts:7-25` и фактически живущий на сервере, а не переписывает его.
8. Миграции — только новые файлы в `supabase/migrations/` (`AGENTS.md`).
