## Blocking findings

### 1. BLOCKER — create flow cannot pass status selection

After a valid Telegram location, `handler.ts:443-456` enters `create_status` and sends the normal `st:*` status keyboard. `callbacks.ts:45` permits `st:*` only while state is `verify_detail`.

Direct pure reproduction at the immutable tip:

```text
validateCallback(create_status, st:green:<current nonce>)
=> {"ok":false,"reason":"wrong_state"}
```

An auditor therefore cannot complete the binding “add verified object” flow. Required: permit the exact status action in the exact create state, route it into `create_facts_steps`, and add a complete handler-level create test from `m:add` through persisted object/photo success.

### 2. P1 — retry after a successful create can delete the committed facade

`places.ts:291-305` uploads with a stable place ID and removes the facade on any RPC failure. Storage uses `upsert: true` at `supabase_stores.ts:275-281`. The DB RPC may succeed, then the success Telegram send or later session/idempotency persistence may fail at `handler.ts:684-690` / `index.ts:124-132`. Because the prior session was not persisted, Telegram retry repeats the same create: upload upserts the same object, RPC fails on the already-existing place ID, and compensation deletes the facade referenced by the committed `photos` row.

Required: make create replay-idempotent across the DB/storage/Telegram boundary. Never delete an object that may pre-exist or belong to a committed place; distinguish ownership/creation by this attempt or make the RPC return success for the same completed operation.

### 3. P1 — an interrupted idempotency claim can remain stuck forever

`supabase_stores.ts:54-70` returns `processing` for every non-completed duplicate. Migration `0002_auditor_bot_sessions.sql:17-21` records `claimed_at`, but no code reclaims an expired claim. If the Edge Function terminates after insert and before the catch/release, all Telegram retries receive HTTP 409 indefinitely.

Required: use an atomic bounded lease/reclaim operation with owner/attempt identity, while still excluding a live concurrent execution. Test process-timeout recovery, not only an in-memory explicit release.

### 4. HIGH — stale buttons from earlier fact steps remain valid

The nonce is constant for the whole flow. `callbacks.ts:48-60` accepts every `sk:*` and every enum callback in every fact state of that flow. Direct reproduction in `create_facts_parking` accepts both the earlier `sk:steps_count` and `rt:none`. `cf:no` is accepted in every state at line 37 as well.

Old keyboards can therefore skip or write the wrong fact after the user advances. Required: bind each action to the exact expected state/step (or rotate a step revision), and test prior-step buttons after every transition.

### 5. HIGH — no complete integration test covers any main state machine

The four handler tests cover `/start`, opening the gray list, one stale nonce, and foreign ownership. None completes gray verification, object creation, or editing through `finalizeFlow` (`handler.ts:608+`). This is why blocker 1 survived a 49/49 green suite.

Required: one handler/store/transport integration test for each main flow, including persisted transitions, outgoing actions, retries, and failure recovery.

### 6. HIGH — real SQL/RPC atomicity remains unproved

The “atomic publish failure” test toggles `publishShouldFail` in `MemoryPlacesStore`; the fake returns before writing. It never invokes `supabase_stores.ts:197` or applies the SQL RPC, so it stays green with a broken RPC name, signature, grant, or rollback.

Required before apply: an isolated/local Postgres integration that applies both T3 migrations, invokes the real store/RPC, proves success, and proves rollback/no partial place-photo pair. Current local DB lint is unavailable, so migration apply remains closed.

## Additional required corrections

### 7. P2 — valid irrelevant Telegram updates are returned as HTTP 400

`webhook.ts:23-24` labels any valid update other than message/callback as unsupported; `index.ts:65-69` responds 400. Without an explicit `allowed_updates` registration constraint, Telegram can retry irrelevant update types. Acknowledge authenticated, structurally valid irrelevant updates with 200/ignored, and constrain webhook registration to `message` and `callback_query` at the eventual gated setup.

### 8. P2 — required auditor secret names are missing from canonical config docs

`env.ts` correctly requires `AUDITOR_TELEGRAM_BOT_TOKEN` and `AUDITOR_TELEGRAM_WEBHOOK_SECRET`, but `.env.example` and the backend/deploy contract do not list their names. Document names only, never values; do not modify the binding backend contract without Alkhas approval.

### 9. P2 — names and aggregate Telegram messages are unbounded

`validation.ts:78-84` trims object names but has no maximum. `handler.ts:752-781` joins up to five database names into one Telegram message without a final payload cap. New or legacy oversized values can exceed Telegram's message limit and make queue delivery fail. Bound names and cap/split final outgoing text.

The branch adds 4,121 lines in 35 files, including an 848-line handler. The existing commits are better than the prior uncommitted handoff, but the handler block still mixes callback routing, all state machines, output formatting, finalize behavior, and persistence coordination.

Required in the correction round: split by natural seams—callback dispatch, message flow, finalize, queue/render—and present focused pushed SHAs. No force-push or history rewrite is authorized.
