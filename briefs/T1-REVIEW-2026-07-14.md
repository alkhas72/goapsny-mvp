## Required fixes

### 1. BLOCKER — required anon photo/RLS proof is vacuous

The binding brief requires executable proof for gray and colored published rows/photos, signed URLs for published facade photos, and denial for hidden/pending parents. The current live database has zero places, zero categories, no `place-photos` bucket, and the branch migrations were not applied. Nevertheless the harness records PASS when rows are absent, skips signed-URL success as PASS, and derives hidden IDs through the anon policy that is itself supposed to hide them.

Evidence:

- `scripts/smoke-public.mjs:144-157` passes gray/colored visibility solely because the query had no error, with `gray_rows=0` and `colored_rows=0`.
- `scripts/smoke-public.mjs:199-219` obtains hidden/pending IDs using the anon client; a correct places policy returns an empty list, so this cannot exercise the photo policy.
- `scripts/smoke-public.mjs:251-266` records published signed-URL success as PASS when no facade row exists.
- Actual run: `published places: rows=0`, `active categories: rows=0`, `Bucket not found`, signed URL `skipped`, final `20/20 PASS`.

Required: run the new migrations in an isolated/local or approved test target with deterministic fixtures owned by privileged setup, then test through a fresh anon client. Include at least gray published + colored published + pending/hidden parents, facade photo objects, and positive/negative signed-URL cases. A skipped required positive case must fail/skip the suite, never count as PASS. Live apply remains an Alkhas gate; do not deploy autonomously.

### 2. BLOCKER — keyboard/focus acceptance is neither implemented nor tested

The brief requires Escape focus return and every control keyboard reachable. Current map pins are Leaflet markers wired only to `click`; the declared `lastMarkerRef` is never assigned, so sheet close cannot return focus to the originating pin. Menu/filter dialogs do not establish or trap focus. Loading/error sheet states have no close control and install no Escape handler because `place` is null.

Evidence:

- `src/components/LeafletMap.tsx:205-220`: markers have click only; no keyboard handler or accessible place name.
- `src/components/PublicMap.tsx:68,127-146,218-229`: `lastMarkerRef` is created and focused but never attached to any interactive marker.
- `src/components/PlaceSheet.tsx:40-54,67-84`: focus/Escape setup exits when `place` is null; loading/error states expose neither close button nor focus target.
- `src/components/MenuOverlay.tsx:15-52` and `src/components/MapFilters.tsx`: modal overlays have no initial-focus, focus containment, or return-focus behavior.
- `scripts/browser-smoke.mjs:43-67`: viewport checks only look for text; no Tab/Shift+Tab, marker activation, Escape, focus-return, sheet states, filters, overflow, or accessibility assertions.

Required: make pins operable and named for keyboard/assistive technology, implement deterministic dialog focus management including loading/error states, and add browser tests that fail before the fix and cover Tab/Enter/Escape/focus return.
