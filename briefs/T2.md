## Final product decision

This is fixed, not a fork:

- Browsing the map is fully anonymous.
- Registration appears only after “Добавить объект”: email → six-digit code.
- Authenticated basic user submits a preliminary gray pin with one facade photo, category, name, and map location.
- The pin is published immediately; no coordinator approval screen or waiting state.
- After successful submission the user cannot edit it; community auditors continue the lifecycle in Telegram.

## Scope B — basic-user gray submission

After OTP, a short mobile flow collects only:

1. one required facade photo;
2. one active category from the approved 12;
3. required trimmed name;
4. location by map center/pin with confirmation; never typed coordinates;
5. summary and “Опубликовать серую метку”.

On success, state explicitly that the gray pin is already on the public map and will be checked by community auditors. Return to/open the same pin. No edit/delete/audit controls for this user.

Do not collect accessibility verdict, measurements, address, comment, private note, extra photos, district, karma, or AI fields.

## Database/security ownership

Add timestamped migrations; never modify existing migrations:

- First migration blocker: `ALTER TABLE public.profiles ALTER COLUMN role SET DEFAULT 'public_user';`. Baseline default `tester` is confirmed to satisfy `current_user_can_collect()` and must not survive before email signup is enabled.
- Make new email-auth profiles `public_user`, with safe trigger/backfill behavior. Do not blanket-update all existing `tester` rows; preserve verified Telegram auditor mappings and backfill only identities proven to be basic email users.
- Add a security-definer RPC `submit_public_place` (exact final name may follow repository convention) matching `OBJECT-CONTRACT.md`.
- The RPC must declare `SET search_path = ''`, fully qualify every table/function reference, revoke default execute, then grant only `authenticated`. Do not copy baseline functions that use `SET search_path = public`.
- RPC input: new `place_id`, name, active category, lat/lng, exact owned facade path. Server hardcodes `gray`, `published`, `public`, `auth.uid()`, and `details.schema_version=1`.
- Validate authenticated `public_user`, UUID/path ownership, uploaded object, active category, finite coordinate bounds, and no existing place.
- Insert `places` and `photos` atomically; grant execute only to authenticated; revoke public/anon.
- Add explicit Storage policies because the baseline collector policy rejects `public_user`: authenticated `public_user` may insert only its own `{place_id}/facade.jpg` in `place-photos`; may delete only the same owned object while no `public.photos.storage_path` references it. No public-user Storage update, extra kind, or cross-owner access.
- Do not grant public users direct place/photo update/delete or any role/status/moderation/audit write.

If the existing profile/auth schema cannot safely distinguish email `public_user` from Telegram auditor, stop for SOL instead of weakening RLS.

## TDD and verification

Write failing tests first for:

- invalid email/no network; request→code; cooldown/double-submit; recoverable wrong/expired token; restored session/sign-out without map regression;
- email user profile is `public_user`, never `tester`/`operator`/admin;
- fresh email identity has `current_user_can_collect()=false`, while a pre-existing legitimate Telegram `tester` is not demoted;
- the new RPC definition has empty search path, fully qualified references, and minimum execute grants;
- public_user Storage insert permits only owned facade path; rejects extra kinds/cross-owner paths; cleanup deletes only the caller's unreferenced facade;
- unauthenticated/anon/elevated-forged caller cannot use the submission RPC;
- missing photo, inactive category, invalid coordinates/path, wrong owner, and duplicate UUID fail without a visible half-object;
- successful RPC writes exact gray/published/public fields and facade row; caller cannot edit afterward;
- gray object is immediately returned by anonymous published query;
- keyboard/focus/error announcement and 44×44 targets for all steps.
