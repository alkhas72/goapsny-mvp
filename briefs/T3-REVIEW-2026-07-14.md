## Findings

1. **BLOCKER — auditor function is wired to the product bot secret.** `supabase/functions/auditor-bot/index.ts:128` reads `TELEGRAM_BOT_TOKEN`; `supabase/functions/auth-telegram/index.ts:94` already consumes the same name for the product Mini App bot, and `.env.example:6` labels it as the GoApsny bot token. In one Supabase project this makes the auditor function reply/download through `@GoApsnyBot`. Rename T3-only env inputs to `AUDITOR_TELEGRAM_BOT_TOKEN` and `AUDITOR_TELEGRAM_WEBHOOK_SECRET`; add missing-env/no-fallback tests and update the report. Never register the auditor webhook on `@GoApsnyBot`.

2. **HIGH — idempotency can both lose updates and double-process concurrent delivery.** `idempotency.ts:10-12` marks the update before processing; `index.ts:113-118` does this before identity/session/handler work, while the catch starts only at `index.ts:144`. A later failure returns 500, but Telegram retry sees “duplicate” and returns 200 without completing the action. Separately, the `hasProcessed` then `insert` sequence in `supabase_stores.ts:54-66` treats a uniqueness conflict as success, so two concurrent requests can both return `new`. Replace it with an atomic claim/completion design; release or make failed work recoverable. Add concurrent-claim and failure→retry tests.

3. **HIGH — verified creation exposes a published half-object.** `supabase_stores.ts:174-200` inserts the place directly as `published`; only afterward does `places.ts:309-315` create the required photo row. Compensation at `places.ts:317-320` removes it later, but anonymous readers can observe the object in between. Storage upload may precede DB work, but place + photo metadata must become visible atomically, via one hardened transaction/RPC or a non-public intermediate state followed by a conditional publish. Test visibility during photo-index failure, not only final in-memory cleanup.

4. **HIGH — gray verification never shows the facade photo.** Selection at `handler.ts:216-248` sends only name/category/coordinates/status prompt. `PlacesStore.getFacadePhoto` exists at `places.ts:23` and `supabase_stores.ts:110`, but has no caller; `TelegramClient` exposes no photo send. The binding flow requires the auditor to inspect the facade before verdict. Add a safe photo delivery path and tests for present/missing media.

5. **HIGH — stale callback buttons are not bound to the current session state.** `handler.ts:176-352` dispatches callback data globally after checking only Telegram user ownership. Old `cat:*`, `st:*`, `sk:*`, or `cf:yes` buttons can mutate/finalize a newer flow; `cf:yes` at `handler.ts:338-340` has no expected-state/nonce check. Bind callbacks to a session version/nonce and whitelist each action for the expected state. Add stale-message and wrong-state callback tests.

6. **HIGH — audit enums are accepted as arbitrary text.** `handler.ts:477-484` directly assigns free text to `ramp_type`, parking/toilet fields, and casts elevator without validation. Invalid table enums fail late; invalid elevator persists in public JSON. Validate all contract enums before state advance and test invalid/recoverable input. Add bounded trimming/length checks for public text fields while touching validation.

7. **MEDIUM — Telegram send failures are swallowed.** `index.ts:43-45` logs a non-2xx `sendMessage` response and returns normally; `answerCallbackQuery` does not check the response. The handler can advance/persist the session and mark the update processed even though the auditor saw no next prompt. Throw a sanitized transport error and cover retry/recovery together with finding 2.

8. **MEDIUM — category migration ownership is duplicated.** T3 adds `0003_sprint_category_toilets.sql`, while T1 already owns `20260714142000_align_active_categories.sql` for the same add/deactivate operation. Keep one canonical migration—T1 by the approved ownership map—and remove the duplicate from T3 before integration.
