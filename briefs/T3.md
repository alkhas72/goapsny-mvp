## Goal

Build the Telegram webhook for authorized community auditors. The auditor verifies an already-public gray pin into green/yellow/red, and can add or edit a verified published object. Moderation is sleeping: normal saves remain public and do not wait for coordinator approval.

## Final product decision

This is binding:

- PWA basic users create gray `published` pins.
- Auditor changes the same object ID from gray to a colored verified verdict and adds audit facts.
- Auditor-created verified objects are `published` immediately.
- Auditor may edit verified published objects.
- `pending`/approval queue is not the normal path; T4 administration may hide/restore.

## Runtime boundary

Create a Supabase Edge Function, not a Mini App UI:

- `supabase/functions/auditor-bot/index.ts` for HTTP/webhook;
- focused Telegram, state-machine, authorization, and persistence modules;
- `[functions.auditor-bot] verify_jwt = false`; webhook secret is the request authenticator;
- additive RLS-enabled server-only session/idempotency table, with no anon/authenticated policies.

Recommended minimum session fields: `telegram_id`, state, draft JSON, `last_update_id`, timestamp. Never store token, unrelated message text, email, or raw photos.

## Security and identity

- Validate `X-Telegram-Bot-Api-Secret-Token` before parsing/processing body.
- Deduplicate Telegram `update_id`; retry cannot duplicate mutation/photo/reply sequence.
- Resolve `from.id` through `profiles.telegram_id`. Allow `tester`/`operator` and `admin`/`owner`; deny unknown, `public_user`, `banned` generically.
- Never accept profile UUID, role, `created_by`, moderation state, or arbitrary Storage path from callback/text.
- Bind callback to the same Telegram identity/session; sanitize logs/errors.
- Service-role and bot token remain only in Edge Function environment.
- Cross-audit invariant: any new/replaced database `SECURITY DEFINER` function must declare `SET search_path = ''` and fully qualify references. Do not copy the baseline `SET search_path = public` pattern. T3 does not take ownership of T1 anon-photo policies or T2 email-role/Storage policies.

## Auditor flows

### A. Verify a public gray pin — P0

1. `/start` identifies the role and offers “Серые метки” and “Добавить объект”.
2. List/page only `status='gray' AND moderation_status='published'`; stable ordering and compact identity/location.
3. Auditor selects a gray object and sees name/category/location/facade plus current facts.
4. Select final status green/yellow/red with word label.
5. Enter or explicitly skip audit facts: steps count/height, door width, ramp, elevator, accessible parking, toilet existence/accessibility, public entrance note, optional address.
6. Review summary, then conditional update of the same ID only if it is still gray+published.
7. Preserve creator/source/history and unknown details; add `details.verification.verified_at` and `verified_by_role='auditor'`. Keep `moderation_status='published'`.
8. Stale/concurrent selection re-fetches actual state and never overwrites another completed audit silently.

### B. Add a verified object — P1

Require facade photo, approved category, name, Telegram location, green/yellow/red, audit facts/explicit skips, and summary. Save as `source='operator'`, `moderation_status='published'`, verification metadata, exact facade path. No coordinator approval step.

### C. Edit a verified object — P1

Authorized auditor may find/select a published colored object and update public audit fields/status/photos. Preserve ID, safe unknown details, and authorship; refresh verification timestamp/role. No delete, private notes, version-history subsystem, or pending patch.

AI autofill, district assignment, karma, external export/links desk research, announcements, and group administration are outside T3.

## Failure behavior

- Never report success until required DB/photo operations have completed.
- New-object photo failure must clean up or remain an explicit server-side recoverable draft, never a visible half-object.
- Gray verification is a conditional atomic update; retry/concurrency cannot revert colored data.
- Expired sessions reset safely to `/start`.
