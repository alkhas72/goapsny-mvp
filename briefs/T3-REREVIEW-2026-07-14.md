## Findings

### 1. BLOCKER — separate auditor bot wiring is still absent

`supabase/functions/auditor-bot/index.ts:79-82,128` still reads `TELEGRAM_WEBHOOK_SECRET` and `TELEGRAM_BOT_TOKEN`. `index_test.ts:5-45` also tests only the old webhook name. `T3-REPORT.md:43,90,109` repeats the old names and even calls the bot token “existing”.

Binding decision: this function belongs only to `@Audit_AIS_Bot` and must read only `AUDITOR_TELEGRAM_BOT_TOKEN` plus `AUDITOR_TELEGRAM_WEBHOOK_SECRET`, with no fallback. `@GoApsnyBot` is the product Mini App bot and its token/webhook must never be reused or changed by T3. Add missing-env/no-fallback tests and update the report.

### 2. BLOCKER — attempted idempotency migration is incompatible with the running store

`supabase/migrations/0002_auditor_bot_sessions.sql:17-21` now requires a non-null `status` with no default. Runtime `supabase_stores.ts:63-66` inserts only `{ update_id }`. Once this migration is applied, the first valid webhook fails at the claim before handler execution.

The original defect is also unchanged: `idempotency.ts:10-12` still does `hasProcessed` then pre-processing `markProcessed`; uniqueness conflict is ignored in `supabase_stores.ts:54-66`, so concurrent requests can both proceed, while any later failure makes a retry look completed. No code marks `completed`, clears/reclaims failed work, or tests concurrency/failure retry.

Required: implement one atomic claim API whose return value distinguishes acquired/in-progress/completed, complete only after durable session/action success, and define safe retry/reclaim behavior. Tests must use a store contract matching the SQL schema and cover concurrent claims plus failure→retry.

### 3. BLOCKER — atomic create migration is broken and unused

`0003_auditor_publish_verified_place.sql` attempts the right transaction boundary but the runtime never calls `auditor_publish_verified_place`: `supabase_stores.ts:164-202` still directly inserts a published place, then `places.ts:309-320` inserts photo metadata afterward and compensates later. Anonymous readers can still observe the half-object.

The attempted RPC has additional defects:

- line 46 uses `p_storage_path <> p_place_id::text || '/facade.jpg'` instead of comparing with the parenthesized concatenated path;
- lines 110-126 revoke default execute from public/anon/authenticated but never grant execute to `service_role`, so the intended caller is not authorized;
- no integration test proves the RPC against an applied schema.

Required: wire the store to one hardened transaction/RPC, correct the path expression, grant only the intended server role, and prove failure atomicity against a real/local Postgres schema. Storage upload may precede the transaction; DB visibility of place+photo metadata must be atomic.

### 4. HIGH — facade photo is still not shown before verdict

`handler.ts:216-248` sends only text when a gray pin is selected. `PlacesStore.getFacadePhoto` remains unused and `TelegramClient` has no `sendPhoto` operation. The auditor therefore cannot inspect the required facade in the verification flow. Add present/missing-photo behavior and tests.

### 5. HIGH — stale callbacks remain globally actionable

`handler.ts:160-352` still checks only Telegram-user ownership. `cat:*`, `st:*`, `sk:*`, pagination and `cf:yes` are not bound to the expected session state or a session nonce/version. An old inline keyboard can alter or finalize a newer flow. Bind callback payloads to the active session revision and whitelist each action by state; add stale/wrong-state tests.

### 6. HIGH — audit enums and public text remain unbounded free text

`handler.ts:450-485` still assigns arbitrary strings to ramp, parking, toilet and elevator fields; public notes/comments have no length bound. Validate every contract enum before advancing, trim/bound public text, and add recoverable-invalid-input tests.

### 7. MEDIUM — Telegram delivery failures are still swallowed

`index.ts:43-45` logs non-2xx `sendMessage` and returns success; `answerCallbackQuery` at lines 47-53 does not inspect the response. A session can advance although the auditor never received the prompt. Throw sanitized transport errors and make them participate in idempotent retry tests.

### 8. MEDIUM — duplicate category ownership and migration version collision remain

T3 still includes `0003_sprint_category_toilets.sql`, although category alignment is T1-owned. It now also includes a second migration beginning with version `0003`, which creates ambiguous/invalid migration ordering. Remove the category migration from T3 and give the atomic RPC a unique timestamp/version coordinated with the final integration order.
