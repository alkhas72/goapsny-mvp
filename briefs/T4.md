# T4 — Telegram coordinator administration

Owner: Composer on Cursor CLI
Worktree/branch: `feat/tg-auditors`, **only after T3 is pushed and SOL records the handoff**
Priority: P2; cut before any PWA-critical work
Mode: **superpowers-full** — privileged authorization/moderation; written plan + TDD + non-author review.
Conductor: SOL; do not edit `run-log.md`.

## Goal

Extend the T3 bot with a minimal coordinator administration contour. Normal gray and verified objects are already public; coordinator approval is not required. Coordinator observes activity, can edit through the trusted audit path, and can hide/restore unsafe or incorrect objects.

## Preconditions

- T3 is pushed/green, handoff is read, and no other executor writes this worktree.
- Read root instructions, absolute `briefs/OBJECT-CONTRACT.md`, and T3 implementation/tests.
- Coordinator is only `profiles.role in ('admin','owner')`.

## Scope

1. Overview: counts of published gray, recently verified/edited, and hidden objects; stable bounded pagination.
2. Object detail with public fields, facade, creator/auditor display label where safe, and timestamps.
3. Hide: conditional `published → hidden`, idempotent, with a short reason stored only in server-side administrative state/log if retained—not public `places.details`.
4. Restore: conditional `hidden → published`, idempotent; re-check required facade and valid object contract before restoring.
5. Edit/verify: route an admin/owner through the same trusted T3 audit logic; verification role may be `coordinator`.
6. Read-only minimal auditor overview from existing profiles and recent activity.

If time is cut, preserve authorization plus hide/restore and their tests. No pending approval queue.

Explicitly out of scope: district/group tables, promoting/demoting profiles, delete, version history, Bitrix, real group/channel administration, announcements, web dashboard, and active pre-publication moderation.

## Concurrency/security

- Every transition includes the expected current state; stale callback re-fetches and shows actual state.
- Only admin/owner sees administrative data/actions. Auditor/basic/unknown roles receive generic denial.
- Restore cannot expose an object missing its required facade or violating the public contract.
- Never expose UUIDs, service-role, raw internal errors, or private identity data in messages.
- Hide/restore preserves object/audit data; no destructive delete.

## TDD and verification

Write failing tests first for:

- tester/operator/public/unknown cannot access admin overview or hide/restore;
- admin/owner overview is bounded, stably ordered, and reports gray/recent/hidden correctly;
- hide changes only published, restore changes only hidden, both idempotent and concurrency-safe;
- restore requires facade/valid public contract;
- hidden disappears from anonymous query and restored returns with same ID/data;
- coordinator edit uses T3 validation and verification metadata;
- no coordinator action is required for a normal public-user gray submission or auditor colored save.

Required evidence:

```bash
deno fmt --check supabase/functions/auditor-bot
deno lint supabase/functions/auditor-bot
deno test supabase/functions/auditor-bot --allow-env
deno check supabase/functions/auditor-bot/index.ts
supabase db lint --local
git diff --check
```

Authorized G2 trace after deployment:

```text
public_user submits gray → anonymous PWA sees same ID gray
→ auditor verifies same ID → anonymous PWA sees colored
→ coordinator hides → absent → coordinator restores → visible again
```

Coordinator steps are recovery/administration, not the publication gate.
